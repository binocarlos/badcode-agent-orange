# Data Provenance & Table Persistence

## When to Use
- Before generating any visualisation or report — check if authoritative tables already exist
- When the user asks where data came from or wants source references
- When building webapps that need "View data" / source drill-down
- When the user wants to save a generated table for future reuse

> **For table access patterns**, see the **table-access** skill. **For report output**, see the **strategic-reports** skill. **For spec discipline (top, side, filter, weight, caseFilter) and the spec summary the user must sign off on**, see the **table-provenance** skill — load it whenever you generate or run a table.

## TOC-First Workflow

**Always check existing tables before generating ad-hoc queries.** Existing TOC tables are authoritative — they have curated specs, descriptions, and may have been validated by the research team.

### Decision flow

1. `pt search "<topic>"` and `pt tables` — look for relevant saved tables
2. If found: use `ask_user` with selectable options to let user pick the right table
3. If not found: generate ad-hoc table, note it as ad-hoc in metadata
4. If modifying an existing table: preserve original filter/weight, confirm changes with user

### Spec preservation

When modifying an existing table, **default to preserving** the original filter, weight, and caseFilter. Ask the user: "Keep the same filter and weight?" — expect YES.

### Why this matters

Ad-hoc tables may use different filters, weights, or code selections than the authoritative version. Using the TOC table ensures consistency with other reports and dashboards.

## Source Metadata

Every visualisation must track its data source. Store source info alongside the output:

### Source types

| Type | Meaning | Example |
|------|---------|---------|
| `toc` | From a saved TOC table | `"path": "Brand/Aided Awareness"` |
| `adhoc` | Generated by agent query | `"spec": {"top": "count(1)", "side": "..."}` |
| `modified` | TOC table with changed filter/weight | `"path": "Brand/Aided", "modifications": ["filter added: Region(1)"]` |
| `composite` | Combined from multiple tables | `"sources": [{"path": "..."}, {"path": "..."}]` |

### Embedding source in webapps

```javascript
// Store source metadata in a data attribute
const chartDiv = document.createElement('div');
chartDiv.dataset.source = JSON.stringify({
  type: 'toc',
  path: 'Brand Tracking/Aided Awareness',
  spec: { top: 'count(1)', side: 'Awareness(cwf\\Total\\;*;nes)' }
});
```

## Click-to-View Data Pattern

**Note:** Platinum dashboards already have built-in click-to-view / drill-down. This pattern is only needed for **standalone webapps**.

When building webapps, include a "View data" button that shows the underlying table:

```html
<button onclick="showDataModal()">View data</button>

<div id="data-modal" style="display:none; position:fixed; inset:0; background:rgba(0,0,0,0.5); z-index:1000;">
  <div style="background:white; margin:5% auto; padding:20px; max-width:900px; max-height:80vh; overflow:auto; border-radius:8px;">
    <div style="display:flex; justify-content:space-between; align-items:center;">
      <h3 id="modal-title">Source: Brand > Aided Awareness</h3>
      <button onclick="this.closest('#data-modal').style.display='none'">&times;</button>
    </div>
    <table id="modal-table" style="width:100%; border-collapse:collapse;">
      <!-- Populated dynamically from embedded data -->
    </table>
  </div>
</div>
```

## Saving Generated Tables

> **NOT YET AVAILABLE** — Saving to TOC requires the TOC write API (#501). For now:

When the agent generates an ad-hoc table the user wants to keep, **prompt by default**: "Shall I save this to your tables folder?" — default YES.

1. Save the spec as JSON: `register_artifact(file_path="saved_spec.json", label="Table Spec", artifact_type="file")`
2. Tell the user the spec can be manually added to the TOC later
3. Include the full spec (top, side, filter, weight) in the artifact

Each user has their own tables folder in the TOC — when the write API is available, tables will be saved there.

## Rules

1. Always check TOC for existing tables before generating ad-hoc queries
2. Present TOC matches to the user via `ask_user` — do not silently pick one
3. Every chart or visualisation must track its source (toc, adhoc, modified, composite)
4. When modifying a TOC table, preserve original filter/weight and confirm changes
5. Include "View data" functionality in webapps that present chart data
6. Note source table path in report narratives (e.g., "Source: Brand > Aided Awareness")
7. When no TOC table exists, note the analysis as ad-hoc and include the full spec for reproducibility
